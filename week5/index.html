TESTING AND DEBUGGING

Exceptions

Trying to call something that is nonexistent will result in a reference error that raises an exception.

Stack Traces: can produce an exception that leads to the point where the error occurred, this is helpful especially if more than one function or method is called.

Warnings

These occur when there's an error in the bode but is isn't enought to cause the program to crash.

Use Strict can be used on a per function basis as well as on the first line of the javascript document.

Linting: Helps to test the qaulity of the code being written beyong just using strict mode. They highlight sloppy programming practices and help make code more consistent and less likely to have problems.

Alert: using alert allows for breakpoints to be placed in the code, it stops a program frim running until okay is clicked.

One of the most useful commands is the debugger keyword which creates breakpoints in your code that will cause the exectuion of the code and allow you to see where the program is currently upto.

Error Objects

An error object can be created by the host enviornment when an exception occurs.

There are seven more error objects used for specific errors:

  • EvalError
  • RangeError
  • SyntaxError
  • TypeError
  • URIError
  • InternalError
  • ReferenceError

The message property returns a description of the error and should be provided as an argument to the Error constructor function. The stack property will return a stack trace for that error. This is a non-standard property and it’s recommended that it is not safe to use in production sites.

Throw - an be applied to any expression causing the execution of the program to stop

Try, Catch, and Finally

If something will result in an exception, wrtap it in a try block.
Ex: function imaginarySquareRoot(number) {
'use strict';
let answer;
try {
answer = String(squareRoot(number));
} catch(error) {
answer = squareRoot(-number)+"i";
} finally {
return `+ or - ${answer}`; }}

Test-driven Development

  1. Write tests (that initially fail)
  2. Write code to pass the tests
  3. Refactor the code
  4. Test refactored code
  5. Write more tests for new features

This is often referred to as the “red-green-refactor” cycle of TDD, as failing tests usually show up as red, and tests that pass show as green.
